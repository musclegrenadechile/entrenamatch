FIX: al agregar un partner en tiempo real (real Firebase mode) no se agregaba / no aparecía en el mapa.

Root cause:
- En modo real, el save hacía setDoc a Firestore pero NO actualizaba el estado local partnerLocations (solo confiaba en el onSnapshot listener).
- Luego hacía setMapForceTick para forzar re-render.
- Pero el listener es async (puede tardar), y además el big map render effect NO tenía en sus deps a showPartners / partnerLocations.length / mapForceTick (solo un pequeño effect que solo hacía invalidateSize).
- Resultado: el nuevo partner se escribía en FS, pero el state del mapa no se actualizaba a tiempo, y el drawing de markers (que está dentro del debounced timeout del big effect) no se re-ejecutaba con los nuevos datos → "no se agrega".

Fix:

CRASH FIX: FirebaseError "Expected first argument to collection() to be a CollectionReference, a DocumentReference or FirebaseFirestore"

Root cause:
- In the partnerLocations listener (added for the "devs add partner businesses on map" feature), we were using `require('firebase/firestore')` inside the useEffect.
- In the Vite production bundle (the index-*.js that runs on github.io and in the webview), mixing `require` with the top-level static imports from services/firebase.ts (which does `getFirestore`) and other dynamic `await import('firebase/firestore')` elsewhere in the app causes the firestore functions to be from a different module instance.
- When `collection(db, 'partnerLocations')` ran, the first arg check inside the (wrong) collection() implementation saw `db` as invalid → crash on map open / partners load.

Fix:
- Changed the partners effect to use the exact same pattern as the rest of the app:
  `const firestoreMod = await import('firebase/firestore'); const { collection, onSnapshot, query, orderBy } = firestoreMod;`
- Kept the `if (!db || isDemoMode)` guard + seeds fallback.
- This makes the listener consistent, avoids duplicate module instances in the bundle, and prevents the type error on collection().

The deprecation warning about enableIndexedDbPersistence() is still there (legacy API); can be cleaned in a future pass by moving to `initializeFirestore(app, { localCache: persistentLocalCache() })` at service init time.

OAuth domain warning is non-fatal info (add musclegrenadechile.github.io to Firebase Auth > Authorized domains for full web Google sign-in on gh-pages).

New AAB + version bump included. The web bundle now has the fixed listener so the collection error should be gone.
